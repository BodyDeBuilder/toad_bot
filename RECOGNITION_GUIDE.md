# Руководство по оформлению команд вкладки «Распознавание»

Документ описывает устройство вкладки **«Распознавание»** модуля мониторинга VK ToadBot, правила оформления новых команд, подразделов и правил, а также конвенции формирования названий подкатегорий и уникальных имён вариантов. Эталонный пример разбора — команда **«Жаба инфо»**.

> [!NOTE]
> Общее описание системы мониторинга/распознавания см. в `instructions.md`. Здесь собраны именно **правила оформления** новых команд — сводные конвенции, чек-листы и шаблоны. Данные хранятся в `data/bot.db`.

---

## 1. Краткая сводка

- Вкладка **«Распознавание»** показывает только те команды из таблицы `monitored_commands`, у которых флаг **`in_recognition = 1`**.
- Иерархия трёхуровневая: **Команда → Подраздел → Правило**.
- Вкладка работает в режиме **Read-Only**: новые правила вносятся только кодом в `src/database/db_manager.py` (раздел «Посев») либо SQL-запросом в БД. UI правил не редактирует.
- Для двух команд («Жаба инфо», «Моя жаба») статус распознавания считается **Python-парсером** (`toad_info_parser.py`), а не regex-правилами из БД. Правила в БД для них носят документирующий/отладочный характер.
- Для всех остальных команд статус считается именно по regex-правилам из `recognition_rules`.
- Жёсткое ограничение (`project_specs.md`, раздел ограничений): **запрещено придумывать команды или regex, не указанные явно пользователем.**

## 2. Трёхуровневая иерархия и схема таблиц

```
monitored_commands          — команда (уникальное имя + флаг in_recognition)
   └── recognition_subcommands   — подраздел (подкатегория): name
          └── recognition_rules       — вариант: pattern, output_value, variable_name
```

Схема таблиц (`src/database/db_manager.py:319-335`):

| Таблица | Поля | Примечание |
|---|---|---|
| `monitored_commands` | `id` PK, `command` TEXT UNIQUE, `in_recognition` INT (0/1), `recognition_status` | Имя команды, флаг показа во вкладке |
| `recognition_subcommands` | `id` PK, `command_id` FK → `monitored_commands` **ON DELETE CASCADE**, `name` TEXT | Подраздел/подкатегория |
| `recognition_rules` | `id` PK, `subcommand_id` FK → `recognition_subcommands` **ON DELETE CASCADE**, `pattern` TEXT, `output_value` TEXT, `variable_name` TEXT | Вариант команды (regex-правило) |

> CASCADE означает: удаление команды удаляет все её подразделы, удаление подраздела — все его правила.

Колонки таблицы правил в UI (`app.js:2247-2249`):
1. **Вариант команды (Regex)** — поле `pattern`.
2. **Что распознаем** — поле `output_value`.
3. **Уникальное имя** — поле `variable_name`.

## 3. Эталонный разбор: команда «Жаба инфо»

### 3.1. Регистрация во вкладке

Посев выполняется при инициализации БД (`db_manager.py:338-345`):

```python
# Посев дефолтных отслеживаемых команд
for cmd_name in ("Моя жаба", "Работа", "Жаба инфо"):
    await conn.execute(
        "INSERT OR IGNORE INTO monitored_commands (command) VALUES (?)", (cmd_name,)
    )
# Принудительно включаем распознавание для «Моя жаба» и «Жаба инфо»
await conn.execute(
    "UPDATE monitored_commands SET in_recognition = 1 "
    "WHERE command IN ('Моя жаба', 'Жаба инфо')"
)
```

Переключение флага `in_recognition` для произвольной команды (через UI кнопку «Добавить команду» / «Удалить из распознавания») идёт через метод `toggle_monitored_command_in_recognition` (`db_manager.py:1704`) и эндпоинт `POST /api/monitor/commands/{id}/recognition/toggle` (`server.py:837`).

### 3.2. Триггер и сопоставление ответа

- **Триггер пользователя** в ядре: `^жаба\s+инфо$` (`knowledge_base.py`, `ACTION_INFO`, регистронезависимо).
- **Сопоставление ответа Жабабота с командой** — в `monitor_manager.py`: по `reply_message`/`fwd_messages` и скользящему окну недавних команд по `peer_id`.
- **Мажоритарное определение**, что ответ — это именно «Жаба инфо»: `match_bot_response` (`knowledge_base.py`). Условие упрощённо: в тексте есть и «покормить», и «откормить», и ещё ≥1 из {работа, подземелье, arena, туса, ограбление}, всего ≥3 категорий → `ACTION_INFO`.

### 3.3. Канонический парсер и вычисление статуса

Для «Жаба инфо» статус «Да/Нет» считается **не** по regex-правилам БД, а функцией `parse_toad_info(text)` из `src/utils/toad_info_parser.py`. Особая ветка — `db_manager.py:1467-1474`:

```python
async def _evaluate_recognition_status(self, conn, command_id, command_name, text, player_vk_id=None):
    if command_name == "Жаба инфо":
        from src.utils.toad_info_parser import parse_toad_info
        parsed_data = parse_toad_info(text)
        if parsed_data is not None:
            if player_vk_id is not None:
                await self.save_toad_state(player_vk_id, parsed_data)  # запись в recognition.db
            return "Да"
        return "Нет"
    ...
```

- `parse_toad_info` построчно разбирает эмодзи-префиксы `💼 🍰 👹 ⚔️ 💃 💘 🥷 🗺` и строку в скобках «(Можно откормить…)».
- Успех возможен **только если присутствуют и распознаны все 9 категорий** (`required_categories`, `toad_info_parser.py:206`). Иначе возвращается `None` → статус «Нет».
- При успехе результат сохраняется в `recognition.db.toad_states` через `save_toad_state` (`db_manager.py:1388`).
- Эндпоинт теста правила: `POST /api/monitor/test-parse` (`server.py:857-974`), спецеветка для «Жаба инфо» на строках 958-961.

## 4. Как формируются названия подкатегорий (`recognition_subcommands.name`)

### 4.1. Правило

- Имя подраздела — **русское**, берётся из **фиксированного набора** (см. `SECTION_KEYS` ниже).
- Соответствие «имя подраздела → ключ состояния/колонка» жёстко задано словарём `SECTION_KEYS` в `src/web/static/app.js:2200-2221`. Этот словарь — единственный источник правды для отображения ключа рядом с названием подраздела.
- Если нужно привязать подраздел к новой колонке состояния — **обязательно** добавьте запись и в БД (посев), и в `SECTION_KEYS`.

## 5. Как формируются уникальные имена вариантов (`variable_name`)

В проекте сложились **три разные конвенции**. Выбор зависит от команды:

### Конвенция А — «Моя жаба»: уникальные теги состояний или имена колонок

Для подразделов с одним вариантом (Имя жабы, Уровень, Сытость, Букашки, Победы, Поражения, Арены) `variable_name` совпадает с именем поля в БД.
Для подразделов с несколькими вариантами (Статус, Состояние, Класс, Настроение) `variable_name` представляет собой **короткий уникальный тег конкретного состояния** (без добавления префикса названия подраздела).

| Подраздел | Вариант (пример) | variable_name |
|---|---|---|
| Имя жабы | Все | `name` |
| Уровень | Все | `level` |
| Сытость | Все | `satiety` |
| Статус | Классик | `classic` |
| Статус | Премиум | `prime` |
| Статус | Премиум+ | `prime_plus` |
| Состояние | Живая | `alive` |
| Состояние | Нужна реанимация | `injured` |
| Букашки | Все | `bugs` |
| Класс | Авантюрист | `adventurer` |
| Класс | Ремесленник | `worker` |
| Класс | Ассасин | `assassin` |
| Настроение | Отличное | `excellent` |
| Настроение | Нормальное | `normal` |
| Настроение | Плохое | `bad` |
| Победы | Все | `wins` |
| Поражения | Все | `losses` |
| Арены | Все | `arenas` |

### Конвенция Б — «Жаба инфо»: английский state-тег, один на вариант, повторяется между подразделами

`variable_name` = **тег состояния** на английском, **один на конкретный вариант** внутри подраздела; **может повторяться** между разными подразделами (например, `ready`/`cooldown` есть и в «Работе», и в «Кормлении», и в «Подземелье»).

Полный алфавитный список тегов, используемых в команде «Жаба инфо»:

`active`, `blocked_by_work`, `claim_pending`, `closed`, `cooldown`, `going`, `home`, `married`, `moving_to_work`, `pending_results`, `preparing`, `ready`, `single`, `working`, `working_at_location`

> Эти теги **точно совпадают** с первым элементом кортежа, который возвращают обработчики в `PATTERNS_*` (`toad_info_parser.py`). Если вводите новый вариант — тег `variable_name` должен быть тем же, что кладёт парсер в `result["<ключ_состояния>"]`.

### Конвенция В — «Работа»: составное описательное имя

`variable_name` — описательное имя на английском в `snake_case`, осмысленное для конкретного варианта: `work_place`, `cooldown_hours`.

### Общее правило выбора конвенции

- Если у команды есть **канонический Python-парсер** («Жаба инфо», «Моя жаба») → `variable_name` обязан совпадать с тем, что парсер кладёт в результат (для Б — это state-тег, для А — имя поля).
- Если парсера нет и статус считается чисто по regex БД («Работа» и будущие команды) → `variable_name` выбирается как осмысленное описательное имя (конвенция В) и должно совпадать с именем колонки/переменной, куда вы хотите записать значение.

## 6. Правила оформления regex (`pattern`)

1. **Именованные группы захвата**: `(?P<имя>паттерн)`. Имя группы = имя подставляемой переменной.
2. **Подстановка в `output_value`**: фигурные скобки `{имя}`. Пример: pattern `... через (?:(?P<hours>\d+)\s*ч:)?\s*(?P<minutes>\d+)\s*мин\.` → output_value `... через {hours}ч:{minutes}мин (working)`.
3. **Флаги движка**: `re.IGNORECASE | re.MULTILINE` (`db_manager.py:1507`).
4. **Автодописывание `$`**: если `pattern` не заканчивается на `$` или `.*`, движок сам допишет `$` в конец (`db_manager.py:1505-1506`). Учитывайте это: по умолчанию правило матчит «до конца строки».
   - В посеве «Жаба инфо» символ `$` в конце паттернов **не ставится** намеренно — его добавит движок.
5. **Экранирование в SQL-строках посева**: обратный слеш удваивается — `\\d`, `\\.`, `\\(`, `\\)`. В «живом» regex это `\d`, `\.`, `\(`, `\)`.
6. **Эмодзи-префиксы**: строки «Жаба инфо» начинаются с эмодзи-маркера категории (`💼 🍰 👚👹 ⚔️ 💃 💘 🥷 🗺`) и пробела/двоеточия — воспроизводите их ровно как в ответе Жабабота. Для «Откорма» маркером является открывающая скобка `(`.
7. **Необязательные группы**: для формата «ЧЧ:ММ» используется `(?:(?P<hours>\d+)\s*ч:)?\s*(?P<minutes>\d+)\s*мин\.` — часы могут отсутствовать. В `output_value` такие группы подставляются как есть; при отсутствии значения подставится пустота.
8. **Альтернативы эмодзи**: `(?:💃🏻|💃)` — учитывает разные кодовые точки (с модификатором тона кожи и без).

## 7. Правила оформления `output_value`

- **Человекочитаемая строка** — то, что показывается пользователю в колонке «Что распознаем».
- **Подстановка значений групп**: `{hours}`, `{minutes}`, `{spouse1}`, `{location}` и т. д.
- **Для конвенции Б («Жаба инфо»)**: в конце строки в скобках указывается **тег состояния** = `variable_name`. Примеры:
  - `Можно покормить (ready)`
  - `Покормить можно через {hours}ч:{minutes}мин (cooldown)`
  - `В браке: {spouse1} и {spouse2} (married)`
- **Для конвенции А («Моя жаба»)**: формат «поле = значение», тег не дублируется: `Имя = {name}`, `Уровень = {level}`, `Сытость = {satiety_cur}/{satiety_max}`.
- **Для конвенции В («Работа»)**: формат «Статус = …, … = …»: `Статус = Работа, Место = {work_place}`.

## 8. Статусы распознавания

Три значения поля `recognition_status` (`instructions.md:114-118`, default `db_manager.py:266`):

| Статус | Когда | Поведение |
|---|---|---|
| `Не распознаем` | Правил/парсера ещё нет; идёт сбор материала | Нейтрально, не влияет на отладку |
| `Да` | Ответ успешно распознан парсером или хотя бы одним regex-правилом | Зелёный бейдж команды во вкладке «Мониторинг» |
| `Нет` | Должен распознаваться, но ни одно правило не сработало | Попадает во вкладку «Отладка»; кнопка «Мониторинг» мигает красным (`pulse-red`) |

Логика вычисления — `_evaluate_recognition_status` (`db_manager.py:1462-1514`):
- «Жаба инфо» / «Моя жаба» → канонический Python-парсер (успех = «Да»).
- Остальные команды → проверка всех `recognition_rules` этой команды; совпадение хотя бы одного = «Да».
- Если у команды вообще нет правил → «Не распознаем».

## 9. Пошаговый чек-лист добавления новой команды во вкладку «Распознавание»

1. **Добавить команду в `monitored_commands`** (если её там ещё нет) и включить распознавание:
   ```python
   await conn.execute("INSERT OR IGNORE INTO monitored_commands (command) VALUES (?)", ("Новая команда",))
   await conn.execute("UPDATE monitored_commands SET in_recognition = 1 WHERE command = 'Новая команда'")
   ```
   Либо через UI: кнопка «Добавить команду» (выбирает из команд с `in_recognition = 0`).

2. **Получить `command_id`** новой команды:
   ```python
   async with conn.execute("SELECT id FROM monitored_commands WHERE command = ?", ("Новая команда",)) as cursor:
       cmd_id = (await cursor.fetchone())["id"]
   ```

3. **Посеять подразделы и правила** в блоке посева `db_manager.py` (рядом с блоками «Моя жаба»/«Работа»/«Жаба инфо», ~строки 348-682). Обязателен guard «сеем только если подразделов ещё нет», как у существующих команд:
   ```python
   async with conn.execute("SELECT COUNT(*) as cnt FROM recognition_subcommands WHERE command_id = ?", (cmd_id,)) as cursor:
       if (await cursor.fetchone())["cnt"] == 0:
           # создаём подразделы и правила
   ```

4. **Шаблон SQL для одного подраздела с правилом**:
   ```sql
   -- Подраздел
   INSERT INTO recognition_subcommands (command_id, name) VALUES (:cmd_id, 'Опыт');
   -- Правило (subcommand_id = ID только что созданного подраздела)
   INSERT INTO recognition_rules (subcommand_id, pattern, output_value, variable_name)
   VALUES (:sub_id, 'получено (?P<xp>\d+) ед\. опыта', 'Опыт = +{xp}', 'xp');
   ```

5. **Если нужен Python-парсер** (как у «Жаба инфо»/«Моя жаба»):
   - Написать функцию `parse_…(text) -> Optional[dict]` в `toad_info_parser.py`.
   - Добавить спецеветку в `_evaluate_recognition_status` (`db_manager.py:1462`) по имени команды.
   - Добавить спецеветку в эндпоинт `POST /api/monitor/test-parse` (`server.py:857`).
   - Ключи результата парсера должны совпадать со значениями `SECTION_KEYS` (см. п. 4.2).

6. **Если новая подкатегория привязывается к колонке состояния** — добавить запись в `SECTION_KEYS` (`app.js:2200-2221`) вида `"Имя раздела": "column_name"`.

7. **Проверить** через `POST /api/monitor/test-parse`, подставив реальный текст ответа Жабабота. Ожидаемый статус — «Да».

8. **Перезапустить сервер** — вкладка «Распознавание» отрисует новые подразделы и правила.

9. **Свериться с ограничением**: команда и regex должны быть явно указаны пользователем (`project_specs.md`). Не придумывать триггеры и шаблоны «по аналогии».

## 10. Жёсткие ограничения и подводные камни

- **Вкладка Read-Only.** Правила вносятся только кодом в `db_manager.py` или SQL в БД. UI их не редактирует (`instructions.md:145`).
- **Запрет на самодеятельность.** Команды и regex должны быть явно указаны пользователем (`project_specs.md`, раздел ограничений). Не вводить «похожие» команды или regex на своё усмотрение.
- **Два парсера `parse_toad_info`.** Канонический — в `toad_info_parser.py` (используется системой статусов и `test-parse`). Legacy-копия — в `src/vk/handlers.py` (не влияет на вкладку «Распознавание»). При правках синхронизируйте оба либо удалите legacy.
- **Правила БД для «Жаба инфо»/«Моя жаба» — документирующие.** Реальный статус «Да/Нет» считает Python-парсер. Если изменится текст ответа — править нужно в первую очередь `PATTERNS_*` в парсере, а уже потом — посев правил в БД.
- **`$` добавляется автоматически.** Не завершайте паттерн на `$`/`.*`, если не уверены, что хотите anchored-матч именно так (по умолчанию движок сам поставит `$`).
- **Экранирование в SQL.** Удваивайте слеши (`\\d`, `\\.`, `\\(`) — иначе SQLite «съест» один уровень.
- **Суффикс `" Вектор"`** в именах подразделов поддерживается кодом, но не используется. Не применяйте без явного указания.
- **CASCADE-удаление.** Удаление команды из `monitored_commands` снесёт все её подразделы и правила. Удаление команды из вкладки (через UI) **не** удаляет правила — оно лишь ставит `in_recognition = 0`.
