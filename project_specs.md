# Техническая спецификация проекта VK ToadBot

Этот документ описывает стек технологий, архитектуру системы и стратегию мобильности (быстрого развертывания и переноса) для мультиаккаунтного бота игры «Жабабот» ВКонтакте.

---

## 1. Стек технологий

*   **Язык программирования**: Python 3.10+
*   **VK API Фреймворк**: [vkbottle](https://github.com/vkbottle/vkbottle) (версия `User` для пользовательских аккаунтов, полностью асинхронный, поддерживает кастомные сессии и LongPoll).
*   **База данных**: SQLite (через асинхронную библиотеку `aiosqlite`) — обеспечивает максимальную мобильность проекта (вся база хранится в одном файле).
*   **Планировщик задач**: [APScheduler](https://apscheduler.readthedocs.io/) (AsyncIOScheduler) — для точного выполнения периодических действий (кормление по кулдауну, отправка на работу, проверка арены).
*   **Скрапинг и парсинг**: `httpx` (асинхронный HTTP-клиент) + `BeautifulSoup4` — для динамического обновления базы знаний с сайта игры.
*   **Управление переменными окружения**: `python-dotenv` — для безопасного хранения системных настроек и токенов.

## 2. Архитектура проекта

Проект строится по модульному принципу. Каждый компонент выполняет строго свою роль:

```
vk_toad_bot/
├── data/
│   └── bot.db                 # Файл БД SQLite (хранит токены, настройки аккаунтов и статистику)
├── config/
│   ├── __init__.py
│   └── settings.py            # Настройки проекта, лимиты, системные константы
├── src/
│   ├── __init__.py
│   ├── database/
│   │   ├── __init__.py
│   │   └── db_manager.py      # Асинхронные методы работы с SQLite (aiosqlite)
│   ├── vk/
│   │   ├── __init__.py
│   │   ├── client_manager.py  # Управление пулом аккаунтов VK (запуск LongPoll для каждого)
│   │   └── handlers.py        # Слушатели сообщений чата (регулярные выражения для детекта хода игры)
│   └── utils/
│       ├── __init__.py
│       ├── parser.py          # Скрипт парсинга базы знаний с сайта toadbot.info
│       └── logger.py          # Настройка красивого логирования логов
├── .env                       # Конфиденциальные данные (загружаются в переменные окружения)
├── .gitignore
├── requirements.txt           # Зависимости Python
├── docker-compose.yml         # Быстрый запуск в Docker
├── Dockerfile
└── main.py                    # Точка входа для запуска бота
```

## 3. Стратегия мобильности и развертывания

Мобильность (возможность быстро отредактировать код, залить его на сервер и сразу продолжить работу без потери данных) — **главный приоритет**. Это реализуется с помощью следующих архитектурных решений:

### А. Сохранение состояния (Persistence)
1. **SQLite в локальном томе**: База данных `data/bot.db` находится в выделенной папке. Она содержит всю динамику: токены аккаунтов, кулдауны действий (когда кормили, когда отправляли на работу), настройки приоритетов и логи.
2. При обновлении кода сам файл базы данных `bot.db` **никогда не перезаписывается и не стирается**.
3. **Переменные окружения**: Все системные ключи хранятся в `.env`, который на сервере создается один раз и не перезаписывается при обновлении кода.

### Б. Сценарии быстрого обновления на сервере

#### Вариант 1: Через Git и PM2 (Самый легкий и быстрый способ)
На сервере устанавливается менеджер процессов `PM2` (он отлично работает с Python-скриптами).
1. Вы вносите изменения локально и делаете `git push`.
2. На сервере запускается короткий скрипт (можно обернуть в bash-скрипт `deploy.sh`):
   ```bash
   git pull
   pip install -r requirements.txt
   pm2 restart toadbot
   ```
3. Бот завершает работу gracefully (закрывает соединения с БД), мгновенно перезапускается с новым кодом, подтягивает состояние из SQLite и продолжает работу LongPoll. Время простоя — **менее 3 секунд**.

#### Вариант 2: Через Docker Compose (Максимальная изоляция)
Идеально для размещения на любом Linux-сервере (VPS) без необходимости настраивать окружение Python.
1. Обновляем код на сервере через `git pull`.
2. Запускаем пересборку:
   ```bash
   docker compose up --build -d
   ```
3. Папка `data/` примонтирована как Docker Volume, поэтому база данных остается нетронутой. Docker пересобирает контейнер за несколько секунд и запускает бота.

## 4. Схема работы бота при перезапускe

1. При старте `main.py` инициализирует `DBManager`.
2. `ClientManager` запрашивает из БД список всех активных аккаунтов и их токены.
3. Для каждого аккаунта создается независимый асинхронный клиент `vkbottle.User` с настроенной сессией API.
4. Запускаются задачи LongPoll: бот начинает слушать сообщения во всех беседах, где состоят эти аккаунты.
5. Запускается `AsyncIOScheduler`, который проверяет по базе данных:
    * Когда в последний раз выполнялась команда «Покормить жабу» для каждого аккаунта.
    * Если кулдаун (12 часов / 6 часов для Prime) прошел, планировщик мгновенно ставит задачу на отправку команды кормления в нужный чат.
    * Если жаба сейчас не на работе, планировщик инициирует отправку на работу.

## 5. Жесткие правила и ограничения разработки

> [!IMPORTANT]
> **Критический приоритет — Только реальные команды пользователя:**
> * В системе категорически запрещено придумывать, дописывать, фантазировать или добавлять «на будущее» любые команды или регулярные выражения, не прописанные пользователем лично.
> * Никакие префиксы (например, префикс «жаба» для команд работы) не должны дописываться или поддерживаться, если на то нет прямой инструкции пользователя. Команды должны распознаваться строго и буквально в том виде, в котором их продиктовал пользователь.
> * Любая добавляемая или модифицируемая логика распознавания (как в Базе Знаний `knowledge_base.py`, так и в других модулях) должна базироваться исключительно на достоверных, явных указаниях пользователя.

## 6. Классификация парсеров (`parser_type`)

Каждая команда в `monitored_commands` имеет колонку `parser_type` (одна из `control` / `simple` / `additive`), которая определяет, **как** её результат должен записываться в `toad_states`. Этот атрибут виден в UI: цветной бейдж справа от имени команды в шапке аккордеона вкладки «Распознавание» (фиолетовый/бирюзовый/оранжевый).

Каждая команда также имеет колонку `description` (TEXT, до 500 символов) — **описание назначения команды**. Это текстовое поле, видимое в UI под именем команды, редактируется inline-кликом. Его назначение:

1. **Для AI-агента:** контекст, помогающий обосновать выбор `parser_type` при добавлении новой команды. Агент читает `description` соседних команд и пишет аналогичное.
2. **Для разработчика:** краткая справка, зачем команда нужна и что она делает, без необходимости лезть в код парсера.
3. **Не участвует в логике** — это чисто информационное поле.

При добавлении новой команды AI-агент **обязан** выбрать тип парсера по критериям ниже, написать описание и явно проставить значения в посеве `monitored_commands`.

### 6.1. Три типа парсеров

| Тип | Код | Цвет бейджа | Суть | Примеры команд |
|---|---|---|---|---|
| **Контрольный** | `control` | фиолетовый | Полный снимок состояния: **перезаписывает все поля** из распарсенного ответа | «Жаба инфо», «Моя жаба», «Мой инвентарь», «Мое снаряжение» |
| **Простой** | `simple` | бирюзовый | Распознаёт только **результат действия** (успех/кулдаун/ошибка) через regex-статус; в БД пишет мало или ничего | «Работа» |
| **Добавочный** | `additive` | оранжевый | Прибавляет/убавляет значения (лут, награды за бой): **инкремент** к существующим | «Покормить жабу», будущая «Напасть на клан» |

В UI вкладка «Распознавание» разбита на **4 подвкладки** по типу парсера: 🟣 Контроль / 🔵 Простой / 🟠 Добавочный / 🟣➕🟠 **Контроль + Лут** (гибриды — команды с `parser_type=control`, которые дополнительно копят инкрементальный лут; ручной список: «Покормить жабу», «Дейлики»). Команды классифицируются функцией `classifyRecCommand()` в `app.js`: гибриды — по списку `HYBRID_COMMANDS`, остальные — по полю `parser_type`.

### 6.2. Критерии выбора типа (для AI-агента)

> Решающее правило читается сверху вниз. Останавливаемся на первом подходящем.

1. **`additive`** — если результат команды — это **прирост или убыток** (`+N` / `-N`), который нужно **прибавить к текущему значению**, а не перезаписать его.
   - Пример: «ты получил 5 🍬» → букашки `+= 5`. Старое значение важно, его нельзя затереть.
   - Парсер возвращает словарь с ключом `_deltas`.

2. **`control`** — если команда возвращает **полный снимок состояния** жабы (сразу много полей: статы, кулдауны, инвентарь), и распарсенные значения должны **затереть** старые.
   - Признак: ответ содержит несколько независимых полей, описывающих жабу целиком.
   - Парсер возвращает «плоский» словарь `ключ → значение`.

3. **`simple`** — во всех остальных случаях, когда от ответа нужно только **понять, что произошло** (успех / кулдаун / ошибка), без детальной записи в `toad_states`.
   - Признак: распознавание идёт по `response_type` (success/cooldown/error) из `response_templates`.
   - Парсер как таковой не нужен — хватает regex-правил БД + `response_type`.

#### 6.2.1. Гибридные команды (состояние + дроп)

Если команда даёт **и полный снимок состояния, и дроп** (например, Дейлики возвращают `daily_status`/`reserve_days` **и** семена в награде):
- **parser_type = `control`** (состояние приоритетнее).
- Дроп кладётся в тот же `_deltas` словарь — `save_toad_state` обрабатывает перезапись (`plain`) и инкременты (`_deltas`) **независимо** в одном вызове.
- Пример: парсер возвращает `{"daily_status": "Золотой", "reserve_days": 5, "_deltas": {"seed_exp_capsule": 1}}`.

### 6.5. Команды-источники лута (additive)

Команды, при выполнении которых могут выпадать **семена** (огород) и **кусочки** (крафт / снаряжение), а также **push-события** Жабабота (арена / клан / получение), копящие результаты боёв и награды. Дроп копится через `_deltas` → `COALESCE(col,0)+N` с аудитом в `delta_audit`.

| Команда | parser_type | Типы дропа | Колонки _deltas |
|---|---|---|---|
| Покормить жабу | additive | крафт-кусочки (🧩🔗🪨🎭📃⚡, +1 рандом), букашки | `cr_*`, `bugs` |
| Откормить жабу | additive | крафт-кусочки (+1 рандом) | `cr_*` |
| Завершить работу | additive | крафт-кусочки (+1…+2) | `cr_*` |
| На Арену | additive | семена (🍭💊🧿), крафт-кусочки (все 6) | `seed_*`, `cr_*` |
| Отправиться в [бронзовое/серебряное/золотое] подземелье | additive | снаряж. кусочки (⚙️🌿🥬🦴), семена (🍭🧿💠) | `eq_parts_*`, `seed_*` |
| Покормить жабенка | additive | снаряж. кусочки (⚙️🌿🥬🦴), семена (🍬) | `eq_parts_*`, `seed_candies` |
| **Бой арены** (push) | additive | исход боя + очки арены + лут из «Ты получил:» | `wins`/`losses`/`draws`, `arena_points`, `bugs`/`cr_*`/`seed_*`/`inv_map` |
| **Клановая война** (push) | additive | общий дроп («Каждый из вас получил:») + индивидуальный («X получил:») | `bugs`/`cr_*`/`seed_*`/`eq_parts_*`/`inv_map` |
| **Получение** (push) | additive | предметы из секции «X получил:» (переводы/подарки) | `bugs`/`cr_*`/`seed_*`/`eq_parts_*`/`inv_map` |

> **Push-команды** (`Бой арены`, `Клановая война`, `Получение`) — это bot-initiated сообщения (см. DATAFLOW.md §3.1). Они не имеют триггера пользователя и резолвятся по тегу `[id|]` / fwd / имени через `AccountResolver` (модуль `src/vk/account_resolver.py`), а не через FIFO-очередь команд. Одно сообщение может адресовать нескольким аккаунтам (особенно клан-бой с общим дропом).

> **Семена** — 6 видов, хранятся в колонках `seed_lollipop`, `seed_bandages`, `seed_gems`, `seed_exp_capsule`, `seed_tape`, `seed_candies`.
> **Крафт-кусочки** — 6 видов, колонки `cr_puzzle` … `cr_lightning`.
> **Снаряжёнческие кусочки** — 4 вида, колонки `eq_parts_weapon` … `eq_parts_beak`.

#### 6.5.1. Источники regex лута — БД «Выпадающий лут»

Рабочие regex-паттерны дропа **хранятся в БД** (`loot_groups` + `loot_items`), а не захардкожены. Это решает проблему частой смены формулировок Жабаботом: правка идёт через UI, без деплоя кода.

- **Одиночные группы** (bugs, arena_points, map): один рабочий regex в `loot_groups.pattern_regex`, предметов нет.
- **Многоэлементные группы** (seed, craft_piece, equipment_part): человекочитаемый шаблон в `pattern_regex`, а рабочий regex — в `loot_items.item_regex` (по одному на эмодзи). Парсер перебирает все предметы группы.
- **Загрузка:** `load_loot_rules_from_db()` при старте формирует `_LOOT_LINE_RULES`. Фолбэк — захардкоженный список, если БД пуста.
- **Привязка к команде** — через `command_loot_config` (см. DATAFLOW.md §2, §7.8). В UI это карточка лута с 5 полями на подвкладках «Добавочный» и «Контроль + Лут».

### 6.3. Правила «ненахода» (что делать, если поле не распознано)

Это ключевая конвенция, единая для всех парсеров. «Ненаход» бывает трёх видов, и поведение строго различается:

| Случай | Поле обязательно? | Поведение в БД | Что пишется в аудит |
|---|---|---|---|
| **Не найдено обязательное поле** | да | **оставить старое значение** (не трогать) | `event_type = missing_required` → предупреждение во вкладке «Аудит дельт» |
| **Не найдено опциональное поле** | нет | **поставить NULL (прочерк «—»)** | `event_type = optional_null` |
| **Поле вообще не упоминалось** | — | **молча оставить как есть** | ничего |

Парсеры сообщают о ненаходе через служебные ключи в возвращаемом словаре:

- `_missing_required: list[str]` — список обязательных полей, которые не нашлись (→ алерт + оставить старое).
- `_optional_nulls: list[str]` — список опциональных полей, которые не нашлись (→ записать NULL/прочерк).
- `_deltas: dict[str, int]` — инкременты для additive-парсера (→ `COALESCE(col, 0) + N`).
- `_command: str`, `_raw_text: str` — метаданные для аудит-следа.

`save_toad_state` (`db_manager.py`) разбирает эти ключи и формирует динамический SQL: для перезаписываемых полей — `SET col = ?`, для дельт — `SET col = COALESCE(col, 0) + ?`, для optional_nulls — `SET col = NULL`. Все события пишет в таблицу `delta_audit`.

### 6.4. Защита от SQL-инъекций

Имена колонок в `save_toad_state` подставляются в SQL динамически (потому что разные парсеры пишут разные наборы полей). Для защиты используется **белый список** `ALLOWED_COLUMNS` (`db_manager.py`). Колонка, не входящая в список, молча игнорируется — её нельзя записать через парсер. При добавлении новой колонки в `toad_states` нужно **обязательно** добавить её и в `ALLOWED_COLUMNS`.
